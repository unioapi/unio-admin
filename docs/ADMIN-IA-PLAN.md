### 1.4 页面壳规范

#### 1.4.1 看板与列表：同一页，不是子菜单（✅ 已定）

各模块 §3.x 改造后的 **默认形态**：

```text
顶栏（标题 + range + 筛选 + 主操作）
  ↓
总览卡（MetricCard 行，0–12 张，因页而异）
  ↓
运维/分析主表（或聚合视图）
  ↓
详情抽屉 / Dialog（点行打开；CRUD 进抽屉或 Dialog）
```

| 问题 | 结论 |
|------|------|
| 看板要不要单独子菜单？ | **❌ 不要** |
| 看板与列表是否同路由？ | **✅ 是** — 同一 URL 纵向堆叠 |
| CRUD 是否单独一页？ | **❌ 不要** — 新建/编辑用 Dialog；复杂配置用抽屉 Tab |
| 例外 | 见 §1.6 看板形态说明 |

**布局组件（实施时）**：`OpsPageLayout` · `QueryCenterLayout` · `ConfigPageLayout` — **仅为 shadcn 组合**，见 §1.5.3。

#### 1.4.2 其它排版规则

| 层级 | 规则 |
|------|------|
| 顶栏 | 侧栏触发 + 页面标题 + 右侧 ≤1 个主按钮 |
| Tab | 子视图 ≥2 时必用；URL `?tab=` 同步 |
| 主区 | 运维页 **不套** 外层大 Card（用 shadcn `Card` 仅用于 Metric 卡片区，见 §1.5） |
| 数字 | 金额 2 位 + Tooltip 全精度 |
| 日期/时间 | **必须** shadcn `Calendar` + `Popover`（或官方 Date Picker 模式），禁止原生 date input 凑合 |

### 1.5 UI 组件与设计规范（shadcn-first，✅ 已定 · **贯穿全计划**）

> **硬约束**：整改与后续所有模块实施 **必须** 遵守本节。验收时违反视为不通过。

#### 1.5.1 核心原则

| 规则 | 说明 |
|------|------|
| **禁止从零自建组件** | ❌ 不得手写 Modal、Dropdown、DatePicker、Tabs、Table 等「新组件」 |
| **必须 shadcn/ui** | ✅ 交互与展示一律基于 `src/components/ui/*`（由 shadcn CLI 安装） |
| **允许二次开发** | ✅ 在 shadcn 组件上 **组合 / 封装**（传 props、拼布局、加业务文案） |
| **缺口先补 shadcn** | 缺组件时 → `shadcn add <component>`，再封装；**不得**用 div+CSS 替代 |
| **设计风格** | 现代化、信息密度适中；**多用**现成 shadcn 块，少裸 HTML |

**项目配置**：`unio-admin/components.json`（style: `radix-nova`，icons: `lucide`）。

#### 1.5.2 禁止 vs 允许

| ❌ 禁止 | ✅ 允许 |
|---------|---------|
| 自建 `<Modal>`、`<Select>` 替代 shadcn | `Dialog` / `Sheet` / `Select` / `DropdownMenu` |
| 原生 `<input type="date">` 作 range 筛选 | `Popover` + `Calendar`（+ 可选 `Button` 触发） |
| 手写 table 样式冒充组件库 | `Table` + `TableHeader` / `TableBody` |
| 纯 div 按钮、无 shadcn 的卡片 | `Button` / `Card` / `Badge` |
| 引入第二套 UI 库（MUI、Ant 等） | 仅 shadcn + Tailwind + lucide |
| 复制 shadcn 源码改到 `components/` 外 | 改 `ui/*` 或通过 wrapper 扩展 |

#### 1.5.3 布局与业务封装（二次开发边界）

以下 **允许** 存在于 `src/components/common/` 或 `src/components/layout/`，但 **内部只能 import shadcn**：

| 封装名 | 组成（shadcn） | 用途 |
|--------|----------------|------|
| `MetricCard` | `Card` + `Tooltip` + `Skeleton` +  typography | §3.x 各页总览卡 |
| `OpsPageLayout` | 页面槽位 + `Tabs`/`Separator` + 筛选区 | 运维页壳 |
| `QueryCenterLayout` | 同上 + 共享 FilterBar | 请求中心三页 |
| `ConfigPageLayout` | `Tabs` + `Card` 分区 | 系统设置 |
| `RequestDetailSheet` | `Sheet` + `Tabs` + `ScrollArea` + `Accordion` | 请求证据中心 |
| `RangeFilter` | `ToggleGroup` 或 `Button` group + `Popover`+`Calendar` | 全部/24H/自定义区间 |
| `FilterBar` | `Input` + `Select` + `Button` + `Badge` | 列表筛选 |

**命名说明**：上述是 **布局/业务模式组件**，不是新的 UI 原语；评审时检查其 render 树是否 100% shadcn 叶子。

#### 1.5.4 模块级组件映射（实施清单）

各整改页 **应优先使用**（已安装见 `src/components/ui/`；缺则 `shadcn add`）：

| 场景 | shadcn 组件 |
|------|-------------|
| 侧栏 | `Sidebar`（已用） |
| 列表 | `Table` + `Pagination` |
| 详情/编辑 | `Sheet`（抽屉优先）或 `Dialog` |
| 表单 | `Field` + `Input` + `Label` + `Select` + `Switch` |
| 状态 | `Badge` |
| 加载/空态 | `Skeleton` + `Empty` |
| 提示 | `Tooltip` + `Alert` + `sonner` Toast |
| 图表 | `Chart`（Recharts 封装，§3.1 趋势） |
| 日期范围 | `Calendar` + `Popover` + `Button` |
| 行操作 | `DropdownMenu` |
| 确认删除 | `AlertDialog` |
| 阈值/设置表单 | `Tabs` + `Input` + `Separator` |

**待安装（实施 IA-0 / 各页 P0 前按需 add）**：`calendar` · `popover` · `toggle-group` · `scroll-area` · `accordion` · `alert-dialog` · `command`（搜索建议可选）。

#### 1.5.5 视觉与现代化

- 遵循 shadcn **radix-nova** 默认间距与圆角；不另起一套 design token。
- 运维表： zebra 可选，用 `TableRow` className；数字列 `tabular-nums` + 右对齐。
- 抽屉宽：`Sheet` `side="right"` + `className="sm:max-w-xl|2xl"`。
- 暗色：沿用现有 `ModeToggle` + CSS 变量，不硬编码色值。
- **多用组件**：能 `Select` 不 `Input` 手输枚举；能 `Tooltip` 不堆脚注小字。

#### 1.5.6 贯穿各实施批次

| 批次 | UI 要求 |
|------|---------|
| **IA-0** | 侧栏仅用 `Sidebar`；计费异常 active 态与 `Tabs` 联动 |
| **D-B / CH-B / …** | 各 `*Page` 改造禁止新增非 shadcn 原语；Metric/Layout 走 §1.5.3 |
| **QX-B** | 全局 `RequestDetailSheet` = `Sheet` 组合 |
| **SYS-B** | 7 Tab = shadcn `Tabs`；诊断工具 = `Card` + `Form` 组件 |
| **验收** | PR 检查：新增 TSX 无「自建交互组件」；date 筛选必须 Calendar |

### 1.8 指标去冗余（✅ 已定，2026-06-23）

> 评审结论：**全部采纳** P0/P1/P2 去冗余建议；下列为实施硬约束，各 §3.x 已同步修订。

#### 1.8.1 全局原则

| 原则 | 说明 |
|------|------|
| **卡片 vs 趋势** | 雷达/总览卡 = **区间快照**；趋势图 = **曲线 only**，不在图内重复大号 KPI |
| **主表 vs 抽屉概览** | 主表 = 全量运维列；抽屉 **概览 Tab** = 基础信息 + 状态/原因 + **最多 2 个摘要指标** + 深链；完整指标在 **性能 Tab** 或主表 |
| **attempt vs request** | **双口径保留**；列头与 Tooltip **必须标注**「attempt 数」或「request 数」，禁止混用 |
| **服务商 vs 渠道聚合** | **保留**两层；服务商页轻聚合，渠道页全量 |
| **毛利率** | **保留**：模型页 8 卡 + 主表 + 抽屉性能 Tab；**移除**：概览分组 Tab、线路抽屉概览 |
| **性能列「一满一瘦」** | 渠道页 **模型 Tab = 完整**（P95/TTFT/TPS）；模型页 **渠道 Tab = 精简**（成功率 + 成本 + 深链） |
| **线路渠道池 Tab** | **线路视角**：命中次数/占比、fallback、深链；性能列 **最多 2 列**（成功率 + P95 延迟），其余链渠道页 |

#### 1.8.2 按页面定稿

| 页面 / 区块 | 定稿 |
|-------------|------|
| §3.1 异常渠道 Top | 渠道 · 健康 · 成功率 · 最近错误 · [查看]（**无** P95/TTFT/TPS） |
| §3.1 分组 Tabs | **Top 5–7 列** 摘要表，非运维全表；线路 Tab **无** 成本/请求/毛利率；模型 Tab **无** 缓存率/毛利率/延迟 |
| §3.3 / §3.5 总览卡 | P95 延迟 + P95 TTFT → 单卡 **「性能」**（Tooltip 展开明细）→ **11 卡** |
| §3.5 渠道池 Tab | 命中/占比 · fallback · 成功率 · P95 延迟（可选）· 深链渠道 |
| §3.5 模型 Tab（抽屉） | 模型 · 可售 · 请求 · 成功率 · 可用渠道 · 深链（**无** 毛利率/全性能列） |
| §3.7 用户抽屉 | 概览 = 余额快照 + range 摘要 + **最近 5 条** 请求/账本 + 深链（**非** 完整指标表） |
| §3.7 项目 / Key 主表 | **移除「成功率」列**（保留请求 + 消费；异常由风险 Badge 表达） |
| §3.8 请求主表 Token | 列显示 `in/out` 紧凑摘要；缓存分项 **仅** 详情 Sheet |

#### 3.1.5 平台状态规则

**三档（宽松版）**

| 档位 | 含义 |
|------|------|
| 正常 | 无明显用户影响 |
| 注意 | 性能/成本/局部渠道变差，平台仍可服务 |
| 异常 | 大面积失败、关键线路不可用、或资金风险 |

**样本保护（✅ 已定）**

```text
最近 15 分钟终态请求数 < 50 → 不直接用百分比判红/判异常；
平台状态卡片显示「数据不足」或维持「正常」并 Tooltip 说明。
```

**异常阈值（15 分钟窗口，命中任一 → 至少「注意」，多项或严重项 →「异常」）**

| 指标 | 异常 |
|------|------|
| 成功率 | < 90% |
| 错误率 | > 10% |
| 超时率 | > 8% |
| P95 Latency | > 30s |
| P95 TTFT | > 12s |
| 线路 | 关键线路连续无可用渠道 |
| 资金 | 高风险计费异常 |

**注意阈值**

| 指标 | 注意 |
|------|------|
| 成功率 | 90% – 97% |
| 错误率 | 3% – 10% |
| 超时率 | 3% – 8% |
| P95 Latency | 15s – 30s |
| P95 TTFT | 5s – 12s |
| 其他 | 缓存读取率明显下降；有渠道降级但线路仍可服务 |

**关键线路（✅ 已定，首版）**

- 定义为：`routes.status=enabled` 且（`is_builtin=true` **或** 至少 1 个 API Key / 项目 default 引用）。
- 「无可用渠道」：该线路在 15 分钟内 routing 失败或无 succeeded attempt（细则见落地计划）。

**超时率口径（✅ 已定，首版）**

- 分子：`request_records.status=failed` 且 `error_code = 'gateway_timeout'`（**仅 gateway 超时**，不含 upstream 其它超时码）。
- 分母：终态请求（succeeded + failed + canceled）。

#### 3.2.8 健康与聚合规则

**渠道健康分桶**：与 §3.3.8 相同（单 channel 维度）。

**Provider 表「健康」列**：展示其下各 channel 的 **四档计数**，不单独发明 provider 健康 Badge（避免与渠道页两套规则）。

**Provider 聚合指标**：

- 请求 / 成功率 / 超时率：该 provider 下所有 channel 的 attempt **合并计数**
- P95 延迟 / TTFT：跨 channel attempt **合并后** 再算分位（非各 channel P95 平均）
- TPS：与 §3.3.9 相同公式，按 provider 汇总 output_tokens / duration

**低样本保护（✅ 已定）**

```text
provider 总 attempt < 20 → 成功率等百分比列显示「样本不足」，不参与「只看异常」的百分比判定。
```

**超时 attempt**：与 §3.3 一致，`error_code = gateway_timeout`。

**「只看异常」**

- 存在不健康或降级 channel **或** 聚合成功率 < 97% 且样本 ≥ 20 **或** 聚合超时率 > 3%。

#### 3.3.8 健康状态规则

**分桶（宽松版，与 §3.1 精神一致）**

| 档位 | 条件（attempt 维度，当前 range） |
|------|----------------------------------|
| 健康 | 成功率 ≥ 97% 且 超时率 < 3% 且 P95 延迟/TTFT 未触发「注意」阈值 |
| 降级 | 成功率 90%–97%，或延迟/TTFT 明显变慢，或超时率 3%–8% |
| 不健康 | 成功率 < 90%，或超时率 > 8%，或连续失败（见下） |
| 无数据 | 区间内 attempt = 0 |

**连续失败（不健康辅助）**

- 最近 N 次 attempt 均 failed（首版 N=5，可配置常量）。

**低样本保护（✅ 已定）**

```text
区间内 attempt < 20 → 不直接用百分比判「不健康」；
显示「无数据」或「样本不足」，Tooltip 说明。
```

**超时 attempt（✅ 已定，首版）**

- `attempt.status = failed` 且 `error_code = 'gateway_timeout'`（与 §3.1 一致；attempt 级后续可扩展 upstream 超时码）。

**「只看异常」**

- 健康 ∈ {降级, 不健康} **或** 价格汇总 ≠ 完整 **或** 凭据未配置。

#### 3.4.1 页面定位

#### 3.4.8 可售 / 健康 / 价格规则

**可用渠道（✅ 已定）**

```text
channel.status = enabled
AND binding.status = enabled
AND 渠道健康 ∈ {健康, 降级}（§3.3.8；无数据/不健康不算可用）
```

**可售模型（✅ 已定）**

```text
model.status = enabled
AND 至少 1 个可用渠道
AND 至少 1 个 enabled binding 上有「有效」channel_prices 售价（input + output 非空且 status 生效）
```

**无可用渠道（告警）**

```text
model.status = enabled AND 可用渠道数 = 0
```

**价格完整（单 binding）**

- 售价：uncached input + output 必填；若业务启用缓存计费则 cache read/write 售价亦必填
- 成本：对应 cost 分项必填
- 主表汇总：任一 enabled binding 缺项 → 列显示最严重缺项

**能力未确认**

- 首版：关键能力 key（stream、tools 等）全部为 `unknown`/缺失，或 catalog 采纳后从未人工确认 → 不可售原因之一

**低样本**

- request 数 < 20：成功率/毛利率等百分比不参与「只看低毛利/异常」判定，显示「样本不足」

**缓存率**

- 与 §3.1 一致：`cache_read_tokens / (cache_read + uncached_input + …)` 按 usage 聚合

**毛利率**

- `(sum revenue_usd − sum cost_usd) / sum revenue_usd`；revenue=0 时显示 `—`

**超时率**（抽屉/性能可选）

- 与 §3.3 一致，request/attempt 维度 `gateway_timeout`

### 3.5 线路（Routes — 路由作战台）

**后端**：routes + route_channels + 客户绑定 + **新增 route 运维/路由聚合**  
**路由**：`/routes`（侧栏「线路」）  
**讨论状态**：✅ **已定**

> **重要性**：高于服务商页。线路是客户真正绑定和使用的 **渠道商品 / 路由策略层**；看板发现成功率低、延迟高、无可用渠道时，往往 **先看哪条线路出问题**，再下钻渠道。

#### 3.5.1 页面定位


|------|------|

| 模型 | 某模型是否可售、渠道覆盖 |

```text

AND 候选池有效（见下）
AND 至少 1 个「可用渠道」（§3.4.8：enabled + 健康/降级 + 有效 binding/价格）
```

**候选池有效**

| 池模式 | 条件 |
|--------|------|
| 全量动态 `pool_kind=all` | 至少 1 个模型存在可用渠道（动态池非空） |
| 手选 `explicit` | `route_channels` ≥ 1 且成员均 enabled |
| 固定单渠道 `fixed`+`explicit` | 恰好 1 条 channel 且 enabled + 健康/降级 |

**异常线路（总览卡计数）**

- 不可服务 **或** 成功率 < 90%（样本 ≥ 20） **或** 无可用渠道次数 > 0 **或** Fallback Rate > 阈值（首版 15%，可配置常量）

**Fallback Rate（✅ 已定）**

```text
有 fallback 的成功 request 数 / 总 request 数
其中「有 fallback」= 同 request_id 存在 attempt_index > 0 且最终 status = succeeded
```

同时展示 **Fallback 次数**（主表列）。

**无可用渠道（✅ 已定）**

- `error_code = routing_no_available_channel`（及同类 routing 失败码）计数，按归属线路聚合。

**成本/请求**

- `sum(cost_usd) / count(requests)`，USD only。

**毛利率（§1.8 移除）**

- ❌ **不在线路页**展示（概览 Tab / 总览卡 / 抽屉概览均不含）。
- ✅ **仅在模型页** §3.4（8 卡 + 主表 + 性能 Tab）保留线路下模型维度毛利率。

**低样本**

- request < 20：百分比类不参与「只看异常」判定，显示「样本不足」。

#### 3.11.6 Tab：阈值规则（**重要**）

**集中管理** 各页引用的健康/异常阈值（§3.1 平台状态、§3.3 渠道、§3.5 线路、§3.4 可售、§3.7 低余额等）。

| 规则组 | 示例 |
|--------|------|
| 平台状态 | 成功率 < 90% → 异常；P95 延迟 > 30s；P95 TTFT > 12s |
| 样本保护 | 最近 15 分钟 request < 50 不判异常（§3.1） |
| 渠道健康 | 成功率 / 超时率 / attempt<20（§3.3.8） |
| 线路健康 | Fallback / 无可用渠道（§3.5.8） |
| 模型可售 | 可售判定相关阈值 |
| 低余额 | 用户 USD 可用 < $5（§3.7） |
| 计费异常 | 触发告警的金额/条数 |
| 缓存率下降 | 概览「注意」规则 |

**UI**：分组表单 + 默认值说明 + **恢复默认**。

**存储（✅ 已定）**
